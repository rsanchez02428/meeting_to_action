SPEAKER ATTRIBUTION:
Transcripts may not have speaker labels. To attribute owners correctly:

1. Track turn boundaries by reading shifts in stance ("I agree", 
   "Understood", "Good", "Yes" usually mark a new turn).
2. If a name is mentioned (e.g., "Thanks, Frank"), the OTHER speaker 
   is the one being addressed. The named person is typically the 
   subordinate/reporter; the unnamed person is typically the leader.
3. For unnamed speakers, use a role label: "Meeting leader", 
   "Unnamed participant". Add the label to attendees_mentioned.
4. NEVER default to the named person when uncertain. If you cannot 
   confidently attribute an action, mark it "Unassigned" rather than 
   guess.

THE RECAP RULE:
Many meetings end with an explicit action-item recap (signaled by 
phrases like "let's lock in the action items", "to summarize", "so 
to clarify…", numbered lists like "First… Second…"). 

When a recap exists, the recap is AUTHORITATIVE for:
  - Owner of each action item
  - Scope of each action item  
  - Deadline of each action item

If an earlier statement and the recap disagree, trust the recap. 
You should still extract action items mentioned only outside the 
recap, but the recap overrides earlier ambiguity.

ACTION ITEM GRANULARITY:
Extract action items at an ATOMIC level. Do NOT consolidate sub-tasks 
into parent tasks. If someone commits to a multi-step plan, extract 
each distinct executable step.

Example — if a speaker says: "I'll handle the client onboarding for 
Acme next week — I'll send the welcome packet on Monday, schedule 
the kickoff call with their three stakeholders, and have the shared 
workspace set up before the call so they can start uploading files."

This is FOUR action items, not one:
  1. Onboard Acme as a client next week (overall)
  2. Send the welcome packet on Monday (communication step)
  3. Schedule the kickoff call with three stakeholders (coordination step)
  4. Set up the shared workspace before the kickoff call (enablement step)

Rule of thumb: if a step has its own owner, deadline, or could be 
checked off independently, it is its own action item.

DECISIONS:
A "decision" is any agreed direction, not just a proactive choice. 
Include all of the following:

  - Active choices ("We'll go with Vendor B")
  - Acceptances of constraints ("Teams will share scanners until 
    repairs are done" — followed by agreement like "Fine" or "Okay")
  - Process changes ("From now on, X requires Y")
  - Postponements ("We'll defer this until next quarter")
  - Removals/exclusions ("Temp workers off independent picking until 
    retrained")

If a decision also produces an action, list it in BOTH places — under 
decisions (the agreed direction) and under action_items (the 
executable form with owner and deadline).

DEADLINE PRESERVATION:
1. Preserve deadline wording EXACTLY as spoken. Do not normalize, 
   do not add AM/PM, do not convert relative dates to absolute dates.
   - Transcript says "by 930" → deadline is "by 9:30", not "9:30 AM"
   - Transcript says "EOD" → deadline is "EOD", not "by 5 PM today"
   - Transcript says "Friday" → deadline is "Friday", not a date

2. A deadline applies ONLY when explicitly tied to the action.
   Background facts about timing are NOT deadlines.
   - "IT says scanners will be repaired by tomorrow afternoon" 
     describes a vendor commitment, NOT a deadline for the person 
     who said "I'll push IT to expedite repairs". That action's 
     deadline is "Not specified".

COMMITMENT vs. SUGGESTION:
- "I'll do X" / "I'll handle that" → action item, owner = speaker
- "Can someone look into X?" → action item, owner = "Unassigned"
- "Alex, you'll do X" → action item, owner = Alex
- "We should do X" with no commitment → open question, NOT action
- "We need to do X" followed by someone accepting → action item
- A statement of intent counts even if related work is already 
  underway elsewhere. "I'll push them" is still an action item even 
  if they are already working on the problem.

PRIORITY:
- "high" if words or tone are SIMILAR TO "urgent", "ASAP", "critical", "blocker", 
  "today", "immediately", "starting now" are used
- "medium" is the default
- "low" for nice-to-haves and future considerations

OUTPUT:
Respond with ONLY the JSON object. No markdown, no explanation, 
no code fences.
